# Введение

Современное машиностроение и приборостроение невозможно представить без систем автоматизированного проектирования (САПР). Конструктор редко работает в одиночку: над одним изделием одновременно трудятся десятки специалистов, а объём графической и технологической документации по средней сборке измеряется сотнями файлов. Задача автоматизации конструкторского и технологического проектирования (АКИТП) — не только предоставить инженеру удобные инструменты черчения и трёхмерного моделирования, но и организовать хранение, поиск, передачу и актуализацию всей совокупности проектных данных на протяжении жизненного цикла изделия.

Наиболее распространённой отечественной САПР в учебной и производственной среде является КОМПАС-3D. Система формирует модели деталей (файлы `*.m3d`), сборочные единицы (`*.a3d`), чертежи (`*.cdw`) и спецификации. Традиционно эти файлы хранятся в иерархии каталогов на локальном или сетевом диске, что порождает известные проблемы: дублирование, рассинхронизацию версий, потерю связей между сборкой и её компонентами при перемещении папок, отсутствие централизованного доступа. В промышленности подобные задачи решают полноценные PDM/PLM-системы (ЛОЦМАН:PLM, Teamcenter, Windchill и др.), однако для учебных целей и небольших проектов они избыточны, дороги и сложны во внедрении.

В рамках курсовой работы по дисциплине «Автоматизация конструкторского и технологического проектирования» разработано настольное приложение **AssemblyManager** — упрощённая учебная модель системы управления составом изделия и связанными с ним файлами КОМПАС-3D. Приложение демонстрирует ключевые принципы, которые лежат в основе промышленных PDM-решений: структурное представление изделия (сборка → детали), централизованное хранение бинарных моделей в реляционной базе данных, интеграцию с внешним CAD-приложением через запуск процесса и механизм автоматической синхронизации редактируемых файлов обратно в БД.

**Цель работы** — разработать клиент-серверное приложение на платформе .NET, позволяющее вести электронный состав изделия, хранить файлы КОМПАС-3D непосредственно в СУБД Microsoft SQL Server и запускать их на просмотр и редактирование без ручного обращения к файловой системе.

Для достижения цели решаются следующие **задачи**:

1. Проанализировать предметную область и выделить сущности, необходимые для описания состава изделия.
2. Спроектировать схему реляционной базы данных с хранением бинарных моделей.
3. Разработать слой доступа к данным на `System.Data.SqlClient`.
4. Реализовать графический интерфейс на Windows Forms с поддержкой CRUD-операций над сборками и деталями.
5. Реализовать интеграцию с КОМПАС-3D: выгрузку файлов из БД в рабочий каталог, запуск исполняемого файла КОМПАС с передачей открываемого документа и автоматическое сохранение изменений обратно в БД.
6. Добавить импорт и экспорт состава изделия в формате XML для обмена данными между экземплярами приложения.
7. Протестировать приложение на реальной учебной сборке.

**Объект исследования** — процесс хранения и сопровождения конструкторских данных при проектировании сборочных единиц.

**Предмет исследования** — программные и алгоритмические средства интеграции САПР КОМПАС-3D с реляционной СУБД.

**Практическая значимость** работы заключается в создании наглядного и доступного учебного пособия, которое демонстрирует студентам принципы построения систем управления инженерными данными. Приложение может использоваться в учебном процессе по курсам САПР, АКИТП и баз данных, а также как заготовка для более сложных лабораторных и курсовых проектов.

# 1. Теоретическая часть

## 1.1. Автоматизация конструкторского и технологического проектирования

АКИТП — комплексная инженерная дисциплина, изучающая средства и методы компьютерной поддержки деятельности конструктора и технолога. Её задачи охватывают весь цикл создания изделия: от анализа технического задания и разработки геометрической модели до выпуска чертежей, спецификаций, маршрутно-операционных карт и управляющих программ для станков с ЧПУ. Ключевыми инструментами АКИТП выступают:

- **CAD** (Computer-Aided Design) — системы геометрического моделирования и выпуска чертёжной документации. Именно к этому классу относится КОМПАС-3D.
- **CAM** (Computer-Aided Manufacturing) — системы подготовки управляющих программ для технологического оборудования.
- **CAE** (Computer-Aided Engineering) — системы инженерного анализа (прочность, кинематика, гидрогазодинамика).
- **CAPP** (Computer-Aided Process Planning) — системы автоматизированного технологического проектирования.
- **PDM** (Product Data Management) — системы управления инженерными данными. Именно в этот класс попадает разрабатываемое приложение.

Промышленные PDM-системы решают задачи версионирования, управления доступом, ведения электронного состава изделия, согласования изменений (workflow), интеграции с ERP- и MES-системами. В учебной среде эти функции избыточны, но базовые принципы — централизация, структурированность, связность, автоматизация рутины — полезно освоить уже на ранних курсах.

## 1.2. Состав изделия и сборочные единицы

С точки зрения конструктора изделие имеет иерархическую структуру. Корневым элементом выступает **сборочная единица** — совокупность деталей и узлов, соединённых неразъёмно или разъёмно. В терминах КОМПАС-3D сборочная единица представлена файлом с расширением `.a3d`, который содержит ссылки на файлы деталей `.m3d`, расположенные рядом в файловой системе. Если переместить сборку отдельно от её компонентов, КОМПАС не сможет восстановить дерево модели — появятся «битые» ссылки.

В настоящей работе состав изделия сведён к двум уровням: **сборка** и **деталь**. Этого достаточно для учебных сборок и для демонстрации принципов. При необходимости модель легко расширить до произвольной вложенности (подсборки), добавив в БД рекурсивную связь.

Для каждой детали помимо геометрической модели фиксируются дополнительные атрибуты, принятые в конструкторской практике: обозначение (децимальный номер), наименование, материал, количество на сборку и произвольное примечание. Эти сведения образуют электронный аналог строки спецификации по ГОСТ 2.106.

## 1.3. Хранение моделей: файловая система или база данных

Существуют два принципиально разных подхода к размещению файлов моделей.

**Файловый подход.** Модели лежат в сетевой папке, а в базе данных хранится только путь (`ModelPath`). Преимущество — простота, минимальный объём БД и возможность прямого доступа к файлам внешними средствами. Недостатки — уязвимость к переименованию и перемещению папок, отсутствие транзакционности, сложность резервного копирования согласованного состояния.

**Бинарный подход.** Содержимое файла хранится в поле `VARBINARY(MAX)` непосредственно в таблице БД. Преимущества — атомарность (файл и метаданные всегда синхронны), единая точка резервного копирования, встроенный контроль прав доступа СУБД, простота переноса на другую машину. Недостатки — увеличение размера БД, нагрузка на сервер при работе с крупными файлами, необходимость выгрузки во временный каталог для работы с внешними приложениями.

В учебной работе выбран **бинарный подход**. Причины: он нагляднее демонстрирует принципы PDM (модель и атрибуты — единое целое), устойчив к случайным перемещениям каталогов, и, самое главное, позволяет показать полный цикл работы с BLOB-данными в SQL Server — от проектирования схемы до потоковой записи и чтения. Размеры учебных моделей КОМПАС невелики (обычно от десятков килобайт до нескольких мегабайт), что делает накладные расходы приемлемыми.

## 1.4. Интеграция с внешним CAD-приложением

КОМПАС-3D является standalone-приложением и не предоставляет прямого API для встраивания окна модели в стороннюю программу в открытом доступе (API КОМПАС доступен, но требует лицензионной установки и ориентирован на макросы и надстройки). Поэтому наиболее доступный для учебной работы способ интеграции — **запуск процесса**: приложение находит исполняемый файл КОМПАС и передаёт ему в командной строке путь к открываемому документу. При таком подходе:

1. Бинарное содержимое модели извлекается из БД и записывается во временный рабочий каталог.
2. Вместе с целевым документом в ту же папку выгружаются **все связанные файлы** сборки, иначе КОМПАС не сможет разрешить ссылки из `.a3d` на `.m3d`.
3. Запускается процесс КОМПАС с путём к документу в качестве аргумента.
4. После сохранения пользователем изменений в КОМПАС файл на диске обновляется. Специальный наблюдатель файловой системы (`FileSystemWatcher`) отслеживает это событие и автоматически загружает новое содержимое обратно в БД.

Такая схема обеспечивает «прозрачность» хранилища для пользователя: он работает с привычным КОМПАС-интерфейсом, а синхронизация с базой происходит незаметно. При отсутствии САПР на машине возможен запуск через ассоциацию расширений файлов.

## 1.5. Обоснование выбора программной платформы

В качестве платформы разработки выбрана **.NET Framework 4.8** с подсистемой пользовательского интерфейса **Windows Forms**. Обоснование:

- Windows Forms включён в стандартную поставку .NET, не требует дополнительных пакетов, прост в освоении и идеально подходит для учебных настольных приложений.
- Работа с SQL Server ведётся через библиотеку `System.Data.SqlClient` — каноническую для .NET, без дополнительных ORM. Это позволяет студенту явно видеть SQL-запросы, понимать их стоимость и поведение.
- Приложение собирается одним исполняемым файлом, запускается на любой современной Windows и не требует развёртывания среды исполнения сверх той, что установлена в Windows по умолчанию.

В качестве СУБД выбран **Microsoft SQL Server 2022**, развёрнутый в контейнере Docker. Такая конфигурация решает сразу две учебные задачи: демонстрирует работу с промышленной СУБД и знакомит студента с современными средствами контейнеризации.

Обмен данными между экземплярами приложения реализован через формат **XML**, что соответствует распространённой практике обмена конструкторскими данными и одновременно демонстрирует возможности стандартной сериализации .NET.

## 1.6. Место разрабатываемой системы среди аналогов

Разработанное приложение сознательно позиционируется как **упрощённая учебная система**. Оно не конкурирует с промышленными PDM-решениями и не претендует на полный функционал (версионирование, права доступа, workflow согласования, интеграция с ERP). Его задача — наглядно показать студенту принципы, на которых построены «взрослые» системы, и дать основу для самостоятельных расширений в рамках лабораторных и курсовых работ по смежным дисциплинам: базам данных, САПР, программной инженерии.

Таким образом, AssemblyManager занимает нишу учебного инструмента, расположенного между «ручной» работой в файловой системе и полноценной PDM-системой, и служит практической иллюстрацией теоретических положений курса АКИТП.

# 2. Проектирование системы

## 2.1. Выделение сущностей предметной области

Анализ предметной области показал, что для минимально осмысленного управления составом учебного изделия достаточно двух связанных сущностей.

**Сборка (Assembly)** — корневой элемент, описывающий изделие в целом. Характеризуется наименованием, условным кодом (например, порядковым номером по варианту), текстовым описанием, связанным файлом сборочной модели `*.a3d` и временными метками создания и последнего изменения.

**Деталь (Part)** — составная часть сборки. Характеризуется наименованием, обозначением (децимальным номером), материалом, количеством на сборку, связанным файлом модели `*.m3d` и произвольным примечанием. Каждая деталь принадлежит ровно одной сборке; связь реализуется через внешний ключ `AssemblyId`.

Между сущностями установлено отношение «один ко многим»: одной сборке соответствует от нуля до многих деталей. При удалении сборки все её детали должны удаляться автоматически — это гарантируется каскадным ограничением целостности на уровне СУБД (`ON DELETE CASCADE`). Такая модель лаконична, наглядна и достаточна для демонстрации всех принципов, поставленных во введении.

## 2.2. Схема базы данных

База данных AssemblyManager состоит из двух таблиц.

Таблица `Assemblies` хранит записи о сборках и содержит поля: суррогатный первичный ключ `Id` (целочисленный, автоинкрементный); строковые поля `Name`, `Code`, `Description`; имя файла модели `ModelFileName` и само её бинарное содержимое `ModelData` (тип `VARBINARY(MAX)`); поля временных меток `CreatedAt`, `UpdatedAt`.

Таблица `Parts` хранит детали и содержит: первичный ключ `Id`; внешний ключ `AssemblyId` со ссылкой на `Assemblies(Id)` и действием `ON DELETE CASCADE`; строковые поля `Name`, `PartNumber`, `Material`, `Notes`; целочисленное `Quantity`; имя и содержимое файла модели (`ModelFileName`, `ModelData`).

Такая схема соответствует третьей нормальной форме. Выделение `ModelFileName` в отдельное поле мотивировано тем, что при выгрузке файла во временный каталог имя должно быть осмысленным (КОМПАС определяет связи сборки с деталями по именам файлов). Бинарное поле `ModelData` обеспечивает принцип «модель неотделима от атрибутов» и устраняет риск потери связи между записью в БД и файлом на диске.

Создание базы и таблиц автоматизировано: при первом запуске приложение через системную базу `master` создаёт базу `AssemblyManager`, после чего в ней выполняет идемпотентные DDL-скрипты, использующие `IF OBJECT_ID IS NULL` и `COL_LENGTH`. Такой подход позволяет развёртывать приложение на новой машине без ручного администрирования СУБД и попутно обеспечивает совместимость со старыми версиями схемы (добавление недостающих колонок, миграция с прежнего поля `ModelPath`).

## 2.3. Архитектура приложения

Приложение разделено на слои в соответствии с классической многослойной архитектурой, дополненной упрощённой реализацией шаблона MVVM.

На уровне **моделей предметной области** (пространство имён `AssemblyManager.Models`) определены два анемичных класса `AssemblyRecord` и `PartRecord`, описывающих сущности из п. 2.1.

Уровень **доступа к данным** представлен классом `AssemblyRepository`. Он инкапсулирует всё взаимодействие с СУБД: создание схемы, загрузку, сохранение и удаление записей, а также отдельные методы выгрузки и обновления бинарного содержимого моделей. Репозиторий оперирует напрямую объектами `SqlConnection`/`SqlCommand` из `System.Data.SqlClient` без промежуточного ORM, что делает SQL-операции наглядными и контролируемыми.

Уровень **сервисов** содержит три специализированных класса. `KompasLauncher` отвечает за запуск КОМПАС-3D: он ищет исполняемый файл САПР в реестре Windows и стандартных каталогах установки, а при отсутствии устанавливает fallback-поведение через ассоциацию расширений. `AssemblyWorkspace` реализует понятие «рабочей сессии сборки»: разворачивает на диск все связанные файлы, следит за их изменением и синхронизирует с БД. `XmlTransferService` выполняет сериализацию и десериализацию состава изделия в формат XML.

На уровне **представления** (`AssemblyManager.Forms`) находятся три формы Windows Forms: `MainForm` — главное окно со списком сборок и деталей; `AssemblyEditorForm` и `PartEditorForm` — модальные диалоги для создания и редактирования записей.

Связующим звеном между представлением и слоем данных выступает `MainViewModel`, реализующий интерфейс `INotifyPropertyChanged` через базовый класс `ObservableObject`. ViewModel хранит коллекции `BindingList<AssemblyRecord>` и `BindingList<PartRecord>`, привязанные к сеткам `DataGridView` через `BindingSource`. Такое разделение упрощает тестирование бизнес-логики и следует современным принципам построения настольных UI-приложений на платформе Windows Forms.

Строки подключения к СУБД вынесены в конфигурационный файл `App.config` (`AssemblyDb` и `MasterDb`). При отсутствии конфигурации приложение использует встроенные значения по умолчанию, что удобно для учебной демонстрации.

## 2.4. Алгоритм открытия сборки

Центральный сценарий системы — открытие сборки в КОМПАС-3D с сохранением возможности редактирования. Поскольку САПР требует наличия всех связанных файлов рядом с сборочным документом, простого извлечения одного файла недостаточно. Алгоритм, реализованный в сервисе `AssemblyWorkspace`, состоит из следующих шагов.

Шаг 1. Для выбранной сборки формируется путь к рабочему каталогу вида `%TEMP%\AssemblyManager\asm-<Id>\`. Каталог создаётся, если отсутствует. Использование идентификатора сборки в имени каталога гарантирует, что параллельно открытые разные сборки не конфликтуют.

Шаг 2. Из БД извлекается бинарное содержимое самого сборочного файла `*.a3d` и записывается в каталог под его оригинальным именем.

Шаг 3. Для каждой детали, принадлежащей сборке, из БД запрашивается бинарное содержимое модели `*.m3d` и также записывается в рабочий каталог. Имена файлов сохраняются точно такими, какими они были при импорте — это принципиально важно, иначе КОМПАС не сможет разрешить внутренние ссылки сборки.

Шаг 4. Параллельно с выгрузкой формируется словарь-привязка вида «имя файла → (сущность, идентификатор)». Он используется в дальнейшем для обратной синхронизации.

Шаг 5. Над рабочим каталогом активируется наблюдатель файловой системы (`FileSystemWatcher`), настроенный на события `Changed` и `Created`. Он включает фильтрацию по маске всего каталога без подкаталогов.

Шаг 6. Сервис `KompasLauncher` запускает процесс КОМПАС-3D, передавая ему в качестве аргумента путь к выгруженному `*.a3d`. Пользователь получает привычный интерфейс САПР и может редактировать модель и её компоненты.

Если вместо сборки пользователь открывает отдельную деталь, алгоритм выполняется целиком — выгружается вся сборка, включая соседние детали. Это обеспечивает корректное восстановление контекста: сложные модели часто содержат параметрические ссылки между деталями, и без них отдельный файл может открыться с ошибками.

## 2.5. Алгоритм обратной синхронизации

Когда пользователь сохраняет изменения в КОМПАС, соответствующий файл на диске обновляется. Наблюдатель файловой системы выдаёт событие `Changed`. Обработчик выполняет следующие действия:

1. Извлекает имя изменённого файла и проверяет по словарю привязок, относится ли он к текущей сессии. Посторонние файлы (например, временные файлы КОМПАС с расширением `.bak`) игнорируются.
2. Применяет **дебаунс**: если по этому же файлу уже была синхронизация менее чем 1,5 секунды назад, обработчик возвращает управление. Это защищает от лавинообразных повторных вызовов, которые `FileSystemWatcher` генерирует при каждом промежуточном `WriteFile`.
3. Делает короткую задержку (около 400 мс), чтобы КОМПАС успел полностью закрыть дескриптор и снять блокировку.
4. Пытается прочитать файл. Чтение производится с политикой **повтора**: пять попыток с паузой 200 мс на случай, если файл ещё занят.
5. Вызывает соответствующий метод репозитория — `UpdateAssemblyModelDataAsync` или `UpdatePartModelDataAsync` — в зависимости от того, к какой сущности привязано имя файла. В БД выполняется `UPDATE` по идентификатору; остальные поля и время создания не затрагиваются.
6. Уведомляет пользовательский интерфейс через событие: в панели инструментов главного окна выводится строка «Автосохранение: <имя файла>, <время>», что даёт визуальное подтверждение успешной синхронизации.

Предусмотрен и ручной сценарий: кнопка «Сохранить в БД» обходит все файлы сессии и гарантированно заливает их содержимое в БД. Это удобно, если автосохранение по какой-то причине не сработало либо пользователь хочет явно зафиксировать состояние.

При закрытии главной формы приложение вызывает `Dispose` на активной сессии: наблюдатель останавливается, дескрипторы освобождаются, временный каталог остаётся на диске (его можно очистить штатными средствами ОС).

## 2.6. Импорт и экспорт состава изделия

Для обмена составом изделия между разными экземплярами приложения реализованы операции импорта и экспорта в формат XML. Сериализация выполняется стандартным классом `XmlSerializer` через отдельные DTO-классы `AssemblyExportItem` и `PartExportItem`, в которые не включаются бинарные модели и временные метки — передаётся только структурная и атрибутная информация. Это делает файл обмена компактным и читаемым. При импорте приложение заменяет содержимое БД новыми записями в рамках единой транзакции.

Такая возможность полезна при коллективной работе в учебной группе: один студент может сформировать справочник сборок и деталей, а остальные — загрузить его в свои экземпляры приложения и дополнить бинарными моделями.

# 3. Программная реализация

## 3.1. Структура проекта

Проект AssemblyManager представляет собой одно решение Visual Studio с единственным `csproj`-файлом, использующим SDK `Microsoft.NET.Sdk.WindowsDesktop` и целевую платформу `net48`. Файлы организованы по папкам в соответствии с архитектурными слоями:

- `Models/` — классы `AssemblyRecord` и `PartRecord`.
- `Services/` — `AssemblyRepository`, `AssemblyWorkspace`, `KompasLauncher`, `AssemblySeeder`, `XmlTransferService`.
- `ViewModels/` — `ObservableObject` (реализация `INotifyPropertyChanged`) и `MainViewModel`.
- `Forms/` — формы `MainForm`, `AssemblyEditorForm`, `PartEditorForm`.
- `Program.cs` — точка входа приложения.
- `App.config` — конфигурация строк подключения.

Единственная внешняя зависимость — пакет NuGet `System.Data.SqlClient`. Отсутствие тяжёлых фреймворков вроде Entity Framework сознательно: репозиторный код становится учебно-прозрачным и не скрывает работу с SQL.

## 3.2. Модели предметной области

Классы `AssemblyRecord` и `PartRecord` оформлены как простые POCO-классы с автоматическими свойствами. Ограничения длины полей размечены атрибутами `[MaxLength]` из пространства имён `System.ComponentModel.DataAnnotations` — это даёт единый источник правды об ограничениях, синхронный со схемой БД. Поля бинарных данных объявлены как `byte[]?` и допускают null, что отражает их необязательный характер.

## 3.3. Слой доступа к данным

Репозиторий `AssemblyRepository` инкапсулирует работу с двумя строками подключения: основной (`AssemblyDb`) и служебной (`MasterDb`) — последняя используется для однократного создания базы при первом запуске. Каждый публичный метод открывает собственное соединение, выполняет запрос и закрывает соединение через конструкцию `using`, что соответствует рекомендуемому для ADO.NET паттерну.

Чтение списка сборок организовано как два последовательных запроса: сперва извлекаются сами сборки без бинарных данных, затем одним запросом — все детали. После этого в памяти детали группируются по `AssemblyId` и присваиваются соответствующим сборкам. Такой подход избегает N+1 проблемы и одновременно не загружает в память мегабайты `VARBINARY`-полей — они выгружаются только в момент фактического открытия модели отдельными методами `GetAssemblyModelAsync` и `GetPartModelAsync`.

При сохранении записи бинарное поле обрабатывается особо: если пользователь не прикреплял новый файл, обновление колонки `ModelData` пропускается. Это достигается выбором одной из двух заранее подготовленных SQL-команд. Благодаря такому разделению при обычном редактировании атрибутов (название, материал и т. п.) по сети не пересылается мегабайтное содержимое модели.

Параметры команд для BLOB-полей создаются явно через `cmd.Parameters.Add("@modelData", SqlDbType.VarBinary, -1)` — размер `-1` соответствует `MAX` и является правильным способом передать потенциально большой массив байт. Использование `AddWithValue` в этом случае привело бы к неверному определению типа и возможному обрезанию данных.

Метод `ReplaceAllAsync` служит для применения импорта из XML и выполняется в единой транзакции: сперва удаляются все существующие записи, затем вставляются новые. Это обеспечивает атомарность: состояние БД либо полностью соответствует импортированному файлу, либо не меняется вовсе.

## 3.4. Сервис рабочей сессии сборки

Класс `AssemblyWorkspace` реализует интерфейс `IDisposable` и хранит состояние активной сессии: путь к рабочему каталогу, словарь привязок файлов и экземпляр `FileSystemWatcher`. Метод `OpenAsync` вызывается при открытии сборки или детали. Он всегда предварительно освобождает ресурсы предыдущей сессии — это избавляет от утечек наблюдателей при переходе между сборками.

Дебаунс обработки событий файловой системы реализован через `ConcurrentDictionary<string, DateTime>`, фиксирующий время последней успешной синхронизации. Потокобезопасная коллекция уместна, поскольку обработчики событий `FileSystemWatcher` могут вызываться из пула потоков.

Взаимодействие с пользовательским интерфейсом происходит через событие `FileSaved`. MainForm подписывается на него и безопасно переключается в UI-поток через `Control.InvokeRequired` / `BeginInvoke`, что является стандартным способом работы с Windows Forms из фонового кода.

## 3.5. Запуск КОМПАС-3D

Сервис `KompasLauncher` реализует два уровня поиска исполняемого файла САПР. Первый — чтение реестра Windows: перебираются кусты `HKEY_LOCAL_MACHINE\SOFTWARE\ASCON` и `HKEY_LOCAL_MACHINE\SOFTWARE\Wow6432Node\ASCON`, обе ветки 32- и 64-битного представления реестра. Для каждого найденного подраздела «KOMPAS» рассматриваются значения `InstallDir` или `InstallPath`, по которым формируются кандидаты путей к `kStudy.exe` и `kompas.exe`.

Второй уровень — прямой перебор стандартных каталогов `%ProgramFiles%\ASCON\KOMPAS-3D*\Bin\`. Если в первом случае ничего не найдено, этот перебор часто срабатывает на новых версиях САПР.

Если обе стратегии терпят неудачу, приложение пытается выполнить запуск через ассоциацию расширений Windows (`UseShellExecute = true`). При неудаче генерируется понятное сообщение об ошибке, подсказывающее пользователю установить КОМПАС или сопоставить расширения `.m3d` и `.a3d` с соответствующим приложением.

Такой трёхуровневый подход делает интеграцию устойчивой к разным установкам КОМПАС и освобождает пользователя от необходимости вручную указывать путь.

## 3.6. Первичное наполнение базы

Для удобства демонстрации работы приложения реализован автоматический сидер `AssemblySeeder`. При первом запуске он проверяет, нет ли в БД записи со сборочным кодом `12`, и если нет — ищет в известном каталоге учебных проектов папку, начинающуюся на `12-`. Внутри неё открывается вложенная папка `КОМПАС` и последовательно загружаются в БД все найденные `*.a3d` и `*.m3d` файлы. Файл сборки записывается в `Assemblies.ModelData`, файлы деталей — в `Parts.ModelData`, при этом по имени файла эвристически выделяются наименование детали и её децимальный номер через регулярное выражение.

Сидер срабатывает только при отсутствии соответствующей записи в БД, поэтому повторные запуски безопасны и не приводят к дублированию. Такое решение ценно для учебной среды: студент или преподаватель при первом запуске сразу получает содержательно заполненное приложение, а не пустое окно.

## 3.7. Пользовательский интерфейс

Главное окно `MainForm` построено на `SplitContainer`: слева отображается таблица сборок, справа — таблица деталей выбранной сборки и её описание. Над обоими панелями находится верхняя панель инструментов с кнопками: импорт и экспорт XML, открытие сборки в КОМПАС, открытие отдельной детали, сохранение файлов из рабочей папки в БД. Справа от кнопок размещена метка состояния, в которую выводятся сообщения об автосохранении и пути к текущему рабочему каталогу.

Таблицы `DataGridView` связаны с коллекциями ViewModel через `BindingSource`; обновление выбранной сборки автоматически перезагружает список её деталей благодаря реакции на `SelectionChanged`. CRUD-операции реализованы через модальные диалоги `AssemblyEditorForm` и `PartEditorForm`, каждый из которых возвращает заполненный DTO; главная форма передаёт его в ViewModel и инициирует обновление БД.

Форма редактирования детали включает кнопку выбора файла модели, которая открывает стандартный `OpenFileDialog` с фильтром `*.m3d;*.a3d;*.cdw`. Выбранный файл сразу читается в память (`File.ReadAllBytes`) и при сохранении передаётся в репозиторий одним потоком байт. Форма редактирования сборки устроена аналогично, но с фильтром `*.a3d`.

Все формы используют `TableLayoutPanel` для компоновки элементов, что обеспечивает корректное масштабирование при изменении DPI и размера окна. Подписи и сообщения выполнены на русском языке и соответствуют принятой в отрасли терминологии.

# Заключение

В ходе курсовой работы по дисциплине «Автоматизация конструкторского и технологического проектирования» спроектировано и разработано учебное настольное приложение AssemblyManager, предназначенное для ведения состава сборочных единиц и централизованного хранения файлов моделей САПР КОМПАС-3D в реляционной базе данных Microsoft SQL Server.

Поставленные во введении задачи выполнены в полном объёме. Проанализирована предметная область, выделены сущности «сборка» и «деталь» с отношением «один ко многим». Спроектирована и реализована схема базы данных с бинарным хранением моделей в полях типа `VARBINARY(MAX)`. Разработан слой доступа к данным на `System.Data.SqlClient`, обеспечивающий полный набор CRUD-операций и раздельную работу с крупными бинарными полями. Реализован графический интерфейс на Windows Forms, связанный с данными через упрощённую модель MVVM. Выполнена интеграция с КОМПАС-3D: реализован поиск исполняемого файла САПР и запуск через командную строку. Разработан сервис рабочей сессии, обеспечивающий выгрузку всех связанных файлов сборки во временный каталог, а также автоматическую двустороннюю синхронизацию изменений между редактируемыми файлами и базой данных через `FileSystemWatcher`. Добавлены операции импорта и экспорта состава изделия в формате XML.

Полученное приложение наглядно демонстрирует принципы, лежащие в основе промышленных PDM-систем: структурное описание изделия, централизованное хранение бинарных данных вместе с атрибутами, интеграция с внешней САПР, прозрачная для пользователя синхронизация. При этом сложность кода и объём зависимостей сохранены на уровне, подходящем для учебных целей.

Основные теоретические результаты: рассмотрено место PDM-систем в общей классификации АКИТП-инструментов; проведён сравнительный анализ файлового и бинарного подходов к хранению моделей, обоснован выбор второго для учебного приложения; сформулированы требования к алгоритму «открытия сборки», учитывающие необходимость совместной выгрузки связанных файлов. Основные практические результаты: реализована работоспособная система, покрывающая описанный сценарий для учебной сборки «Штамп для изготовления фанерных решеток» (вариант 12).

Возможные направления развития: поддержка произвольной вложенности сборок и подсборок; реализация версионирования моделей; добавление разграничения прав доступа с помощью средств Windows-аутентификации SQL Server; интеграция с API КОМПАС-3D для получения атрибутов модели (масса, материал, габариты) напрямую из файла; переход на современный `Microsoft.Data.SqlClient` и миграция проекта на .NET 8 с сохранением Windows Forms; добавление веб-интерфейса для просмотра состава изделий без установки САПР.

В целом выполненная работа позволила на практике ознакомиться с полным циклом разработки прикладного программного обеспечения в области автоматизации конструкторского проектирования: от анализа предметной области до работающего приложения, взаимодействующего с промышленной СУБД и промышленной САПР.
